# MiroFish v4.0 — 多代理辩论 + 预测引擎

**版本：** v4.0
**更新日期：** 2026-04-06
**核心哲学：** 不追求长度，追求准确。每个预测都可验证，每个分歧都用 DB 事实裁决。

---

## 0. 设计动机

### 从 MiroFish 原始项目 issues 学到的

| Issue | 痛点 | v4 解决方案 |
|-------|------|-------------|
| "可以预测股票吗？搞一个未来价格预测吧，这种泛泛而谈的算命好像没什么用" | 报告空洞，无具体预测 | 每个情景输出可验证预测 + 触发条件 + 时间线 |
| 发帖数量过少（几百 agent 循环 80 次只有十几条留言） | 仿真引擎质量差 | 轻量多代理辩论 — 不仿真数量，保证每次交互质量 |
| Agent 缺乏心理深度 (issue #472) | 6 个角色是同一个人设 | 每个角色有独立背景/认知偏差/利益驱动 |
| 预测到执行的闭环缺失 (issue #399) | 分析完就结束了 | 预测 → 记录 → 追踪准确度 → 持续校准 |

### 核心原则

1. **准确 > 长度** — 3000 字但预测准确的报告 > 7000 字但空洞的分析
2. **可验证 > 合理** — 每个结论必须能用 DB 数据或未来事实验证
3. **分歧是资产** — 不同角色的冲突观点是信号，不是噪音
4. **DB 是仲裁者** — 当角色分歧时，用 8 个本地 DB 的事实数据裁决
5. **校准 > 自信** — 记录每次预测的准确度，持续修正模型

## 1. 报告结构

### 标题格式
`MiroFish_[Topic]_v4_[YYYYMMDD].md`

### 必需章节（顺序不可变）

| 序号 | 章节 | 核心要求 |
|------|------|----------|
| 0 | **预测摘要** | 3 个最重要的可验证预测，含触发条件和时间线 |
| 1 | 输入与假设 | 数据来源、关键假设、不确定性范围 |
| 2 | **多代理辩论记录** | 6 个角色的独立立场 + 交叉挑战 + DB 仲裁 |
| 3 | **共识与分歧清单** | 明确列出哪些达成一致，哪些仍有分歧 |
| 4 | **情景预测** | 3 个情景，每个含概率、触发条件、验证时间、可观察指标 |
| 5 | **预测记分卡** | 过去预测的准确度回顾，校准当前模型的偏差 |
| 6 | 行动建议 | 基于共识情景的具体行动，含止损线 |

> 不再有：Entity Extraction（信息密度低）、KPI Verification Log（形式化但无用）、Personal Positioning（冗余）

## 2. 多代理辩论（核心创新）

### 6 个代理的定义

每个代理不是"角色模板"，而是有独立背景、认知偏差和利益驱动的**独立分析者**。

| 代理 | 背景原型 | 认知偏差 | 关注焦点 | DB 数据偏好 |
|------|----------|----------|----------|-------------|
| **量化分析师** | 对冲基金 Quant | 过度依赖历史数据，低估黑天鹅 | 资金流向、波动率、统计信号 | market_data, 13F |
| **周期主义者** | 宏观策略师 | 过度拟合历史周期 | 历史对比、周期位置、领先指标 | news.db, research_reports |
| **基本面投资者** | 价值投资者 | 低估市场非理性持续时间 | 财务指标、估值、竞争壁垒 | company_fundamentals, SEC |
| **叙事交易者** | 社交媒体驱动 | 易受叙事驱动，忽视估值 | 市场情绪、叙事可信度、催化剂 | important_persons, news |
| **地缘分析师** | 国际关系专家 | 过度关注地缘风险 | 政策、地缘冲突、供应链 | news.db, evidence_sources |
| **风控官** | 首席风险官 | 过度保守，低估上行空间 | 尾部风险、流动性、杠杆 | 所有 DB，重点找矛盾 |

### 辩论流程

```
Phase 1: 独立立场
  → 每个代理基于自己的视角和 DB 数据，给出独立判断
  → 每个判断必须包含至少 1 个 DB 引用和 1 个可验证预测

Phase 2: 交叉挑战
  → 代理之间互相挑战对方的论点
  → 格式："量化分析师 质疑 叙事交易者: [具体质疑]"
  → 被质疑方必须回应

Phase 3: DB 仲裁
  → 当两个代理的论点矛盾时，查询 DB 数据裁决
  → 格式："DB 仲裁: [查询结果] → 支持 [代理名]"
  → 如果 DB 数据不足，标记 "⚠️ 数据不足，无法裁决"

Phase 4: 共识提取
  → 列出所有代理达成一致的观点
  → 列出仍存在分歧的观点
```

### 辩论输出格式

```markdown
## 多代理辩论记录

### Phase 1: 独立立场

**量化分析师**: [基于 market_data 和 13F 的独立判断，含 1 个可验证预测]

**周期主义者**: [基于 news.db 历史对比的判断，含 1 个可验证预测]

...（6 个代理）

### Phase 2: 交叉挑战

**量化分析师 质疑 叙事交易者**: "你说 AI 应用将轮动上涨，但 13F 数据显示 BlackRock 和 Citadel 过去两季都在减持软件股。解释这个背离。"

**叙事交易者 回应**: "[回应 + DB 引用]"

**DB 仲裁**: `company: MSFT` → PE 18.2x，4 年低位；`13f: MSFT` → BlackRock 持有量 -8%。数据部分支持量化分析师（确实减持），但 PE 低位也支持叙事交易者（估值合理）。→ ⚠️ 双方都有数据支撑

### Phase 3: 共识与分歧

**共识**:
1. [观点 1] — 所有代理同意
2. [观点 2] — 5/6 代理同意，仅风控官有保留

**分歧**:
1. [观点 A] — 量化分析师 vs 叙事交易者（DB 数据矛盾，⚠️ 无法裁决）
2. [观点 B] — 周期主义者 vs 基本面投资者（分歧在于时间窗口）
```

## 3. 预测引擎

### 可验证预测的格式

每个预测必须包含 5 个字段：

```markdown
| # | 预测 | 触发条件 | 验证时间 | 当前概率 | 验证状态 |
|---|------|----------|----------|----------|----------|
| 1 | [具体可观察事件] | [什么发生时视为命中] | [何时验证] | [X%] | 待验证 |
| 2 | [具体可观察事件] | [什么发生时视为命中] | [何时验证] | [X%] | 待验证 |
```

### 预测质量规则

| 规则 | 说明 | 反面示例 |
|------|------|----------|
| 具体 | 必须能用二元判断（是/否）验证 | ❌ "市场可能波动" → ✅ "标普 500 在 30 天内跌破 6100" |
| 可观察 | 结果公开可查，无需主观判断 | ❌ "情绪转好" → ✅ "VIX 在 14 天内跌破 20" |
| 有时间线 | 明确的验证时间窗口 | ❌ "未来某时" → ✅ "2026 年 Q2 结束前" |
| 有概率 | 每个预测附带主观概率 | ❌ "一定会" → ✅ "60% 概率" |

### 情景预测格式

```markdown
## 情景预测

### 情景 A: 基准情景 (50%)
- **描述**: [1-2 句话描述]
- **触发条件**: [具体的、可观察的条件]
- **时间窗口**: [预计持续或验证时间]
- **关键指标**:
  - [指标 1]: 预期值 [X]
  - [指标 2]: 预期值 [Y]
- **可验证预测**:
  1. [具体预测 1] → 验证时间: [日期]
  2. [具体预测 2] → 验证时间: [日期]

### 情景 B: 悲观情景 (25%)
...

### 情景 C: 乐观情景 (25%)
...

> 概率总和 = 100%
```

## 4. 预测记分卡（v4 新增）

### 目的

记录过去预测的准确度，回答核心问题：**这个分析系统的预测到底准不准？**

### 格式

```markdown
## 预测记分卡

### 历史预测回顾

| # | 预测内容 | 预测日期 | 概率 | 验证日期 | 实际结果 | 是否命中 | 校准偏差 |
|---|----------|----------|------|----------|----------|----------|----------|
| 1 | [预测 1] | 2026-03-15 | 70% | 2026-03-30 | [实际发生] | ✅ | -5% (偏乐观) |
| 2 | [预测 2] | 2026-03-15 | 30% | 2026-04-01 | [未发生] | ✅ | 准确 |
| 3 | [预测 3] | 2026-03-20 | 80% | 2026-04-05 | [未发生] | ❌ | +50% (严重偏乐观) |

### 校准统计

- **总预测数**: [N]
- **命中率**: [X/Y, Z%]
- **Brier Score**: [0-1，越低越好]
- **校准偏差**: [预测概率 vs 实际命中率的差距]
- **最近 5 次命中率**: [X/5]
- **整体评级**: [可靠 / 偏乐观 / 偏悲观 / 不可靠]
```

### 数据来源

1. **新报告**: 生成报告时自动记录到 `data/predictions/` 目录
2. **历史 Obsidian 报告**: 扫描 `Reports/` 目录中的旧报告，提取过去的预测和当前结果
3. **校准文件**: `data/predictions/scorecard.json` 维护累计统计

### Brier Score 计算

```python
brier_score = mean((predicted_prob - actual_outcome) ** 2)
# actual_outcome = 1 if happened, 0 if not
# 完美预测 = 0, 随机猜测 = 0.25 (对二元预测)
```

### 校准偏差

```
校准偏差 = mean(predicted_prob) - mean(actual_outcome)
> 0: 系统性偏乐观（预测概率高于实际命中率）
< 0: 系统性偏悲观（预测概率低于实际命中率）
= 0: 完美校准
```

## 5. 数据源与可信度

### 8 个本地 DB

| DB | 可信度 | 使用原则 |
|----|--------|----------|
| SEC filing 原文 | ★★★★★ | 法定披露，直接引用 |
| company_fundamentals | ★★★★☆ | XBRL 结构化数据 |
| 13F 机构持仓 | ★★★★☆ | SEC 法定披露 |
| news.db (importance≥5) | ★★★☆☆ | 经重要性过滤 |
| market_data.db | ★★★☆☆ | 日线级，非实时 |
| important_persons.db | ★★★☆☆ | 507 位人物，需交叉验证 |
| research_reports.db | ★★★☆☆ | 学术论文，高质量但可能过时 |
| evidence_sources.db | ★★☆☆☆ | 来源可信度元信息 |

### DB 仲裁规则

当代理之间分歧时：
1. 查询相关 DB
2. 如果数据明确支持一方 → 标记 "DB 支持"
3. 如果数据矛盾或不足 → 标记 "⚠️ 数据不足，保留分歧"
4. **永远不强行制造共识** — 保留分歧本身就是有价值的信号

### Obsidian 历史报告的角色

不再标注为"可能含偏见"，而是作为：
1. **预测校准记录** — 提取过去的预测，对比实际结果，更新记分卡
2. **思维参考** — 过去的分析框架和逻辑链（非结论）
3. **偏差追踪** — 如果历史报告反复在某个方向犯错（如过度乐观），在系统 prompt 中标注

## 7. 生成流程

```
Step 1: 收集输入
  - 读取视频转录 / 主题描述
  - 运行 compile_brief_v4() 获取 8 DB 快照
  - 搜索 Obsidian 历史笔记
  - 加载预测记分卡

Step 2: 多代理辩论
  - Phase 1: 每个代理独立发表立场（各含 1+ DB 引用和 1 可验证预测）
  - Phase 2: 代理互相挑战
  - Phase 3: DB 仲裁分歧
  - Phase 4: 提取共识与保留分歧

Step 3: 情景预测
  - 基于共识和保留分歧，生成 3 个情景
  - 每个情景含 2-3 个具体可验证预测

Step 4: 更新记分卡
  - 回顾历史预测准确度
  - 计算 Brier Score 和校准偏差
  - 如有系统性偏差，在建议中调整

Step 5: 行动建议
  - 基于共识情景和校准后的偏差，生成建议
  - 每条含触发条件、止损、验证方法

Step 6: 记录
  - 将新预测写入 data/predictions/scorecard.json
  - git commit
```

## 9. 预测数据存储

### 文件结构

```
data/predictions/
├── scorecard.json          # 累计预测统计
├── predictions.db          # 所有预测记录 (SQLite)
└── reports/                # 每份报告的预测提取
    └── [YYYYMMDD]_[topic].json
```

### predictions.db 表结构

```sql
CREATE TABLE predictions (
    id TEXT PRIMARY KEY,
    report_id TEXT,
    prediction TEXT,              -- 具体预测内容
    trigger_condition TEXT,       -- 触发条件
    predicted_prob REAL,          -- 预测概率 (0-1)
    predicted_at TEXT,            -- 预测日期
    verify_by TEXT,               -- 验证时间
    actual_outcome INTEGER,       -- 1=命中, 0=未命中 (验证后填入)
    verified_at TEXT,             -- 验证日期
    agent TEXT,                   -- 哪个代理提出的
    tags TEXT                     -- 标签: [geopolitics, market, ...]
);
```

### 自动验证

```bash
# 定期检查待验证的预测是否已有结果
python scripts/verify_predictions.py

# 查看当前记分卡
python scripts/scorecard.py
```

## 10. 与 v3 的关系

| 维度 | v3.0 | v4.0 |
|------|------|------|
| 核心结构 | 6 角色单作者报告 | 6 代理独立辩论 |
| 输出重点 | 完整分析（长篇） | 可验证预测（精准） |
| 分歧处理 | 隐藏 | 显式标注 + DB 仲裁 |

**v4 不是 v3 的加长版，而是 v3 的精确化。**

## 11. 多主题支持 (v4.1)

### 11.1 设计动机

原始 v4 spec 仅面向金融/投资分析，但实际内容中约 48.6% 为非金融主题（社会评论、战争冲突、旅行生活方式等）。v4.1 扩展系统根据视频主题自动切换分析框架。

### 11.2 主题分类

系统根据视频标题和字幕内容自动识别主题类别，不同主题使用不同的分析配置：

| 类别 | 适用场景 | 代理团队 | 行动建议 |
|------|----------|----------|----------|
| 金融/投资 | 股票、宏观、加密货币、能源 | 量化/周期/基本面/叙事/地缘/风控 | 仓位/止损/对冲 |
| 战争/冲突 | 战区报道、冲突分析、撤离叙事 | 战地记者/地缘/国际法/人道主义/安全/历史 | 安全路线/风险评估 |
| 社会生活 | 社会现象、文化评论、移民体验 | 社会学家/文化观察者/媒体分析/经济/心理/历史 | 趋势理解/观点参考 |
| 旅行/生活方式 | 旅行体验、数字游民、避战旅居 | 旅行作家/当地通/安全顾问/预算/文化桥梁/风险官 | 行程建议/实用信息 |
| 科技 | 技术趋势、产品分析、AI 产业 | 技术分析师/行业观察/投资人/UX/竞争分析/风险官 | 技术判断/投资参考 |

### 11.3 分类机制

主题分类基于标题 + 前 3000 字符字幕的关键词权重匹配：
- 每个关键词有优先级权重（如"战火"=20, "导弹"=20, "逃离"=18 > "冲突"=8, "中东"=8）
- 解决关键词冲突：如"逃离中东战火"→战争/冲突而非地缘政治
- 无匹配关键词时默认为金融/投资（向后兼容）

### 11.4 数据源控制

非金融主题自动跳过无关金融数据源（FRED、SEC、13F、估值、波动率、能源等），仅保留：
- news.db（通用新闻搜索）
- livenews（实时快讯）
- important_persons.db（相关人物）
- evidence_sources.db（来源可信度）
- Tavily 在线搜索

### 11.5 通用框架

无论何种主题，以下结构保持一致：
- 多代理辩论的 4 个阶段（独立立场 → 交叉挑战 → DB 仲裁 → 共识提取）
- 可验证预测的格式（具体、可观察、有时间线、有概率）
- 共识与分歧的显式标注
- 预测记分卡（Brier Score 校准追踪）

### 11.6 扩协议

添加新主题类别仅需：
1. 在 `topic_config.py` 中添加 `TopicCategory` 枚举值
2. 定义 `AgentProfile` 列表（6 个代理）
3. 定义 `natural_tensions`（代理间张力对）
4. 定义 `TopicConfiguration` 并注册到 `TOPIC_CONFIGS`

无需修改 `debate_engine.py`、`prediction_store.py` 或核心辩论流程。
